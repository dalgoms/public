# 에러 수정 및 개선 히스토리

이 문서는 MEFIMAKE 개발 과정에서 발생한 주요 버그, 솔루션, 그리고 배운 교훈을 기록합니다.

---

## 1. 선택 시스템 버그

### 문제

**날짜:** 2026년 2월  
**심각도:** 높음  
**영향:** 사용자가 배경을 콘텐츠와 별도로 제어할 수 없음

**증상:**
캔버스 배경을 클릭하면 엘리먼트로 선택되어, 사용자가 실수로 프로퍼티 패널을 통해 배경을 이동하거나 수정할 수 있었음.

**근본 원인:**
선택 이벤트 핸들러가 배경 엘리먼트와 콘텐츠 엘리먼트를 구분하지 않았음. 캔버스 내의 모든 엘리먼트가 선택 가능했음.

```javascript
// 문제가 있는 코드
canvas.addEventListener('click', (e) => {
    selectElement(e.target);  // 모든 엘리먼트 선택
});
```

### 솔루션

**배경 격리 규칙** 구현 - 배경 엘리먼트는 선택에서 명시적으로 제외됨.

```javascript
// 수정된 코드
canvas.addEventListener('click', (e) => {
    const target = e.target;
    
    // 가드: 배경 엘리먼트는 절대 선택하지 않음
    if (target.classList.contains('ad-container') ||
        target.classList.contains('ad-overlay') ||
        target.classList.contains('ad-background-image')) {
        deselectAll();
        return;
    }
    
    // 콘텐츠 엘리먼트만 선택
    if (target.classList.contains('ad-headline') ||
        target.classList.contains('ad-subtext') ||
        target.classList.contains('ad-cta')) {
        selectElement(target);
    }
});
```

### 배운 교훈

**원칙: 명시적 엘리먼트 분류**

모든 클릭 가능한 엘리먼트가 선택 가능해야 한다고 가정하지 말 것. 명확한 카테고리(배경 vs 콘텐츠)를 만들고 사용자 의도에 따라 다르게 처리.

## 2. 배경 오버라이드 버그

### 문제

**날짜:** 2026년 2월  
**심각도:** 중간  
**영향:** 배경 이미지 불투명도 컨트롤이 작동하지 않음

**증상:**
배경 컨트롤의 불투명도 슬라이더가 배경 이미지에 효과가 없었음. 슬라이더 위치와 상관없이 이미지가 완전 불투명 상태 유지.

**근본 원인:**
불투명도 제어에 사용되는 `.ad-overlay` 엘리먼트가 단색 대신 `background: transparent`를 사용함. 불투명도를 조정해도 시각적 효과가 없었음.

```css
/* 문제가 있는 CSS */
.ad-overlay {
    position: absolute;
    background: transparent;  /* 불투명도를 보여줄 색상 없음! */
    opacity: 0;
}
```

### 솔루션

오버레이 배경을 단색 검정으로 변경한 후, 불투명도를 조절하여 배경 이미지 위에 어둡게 하는 효과 생성.

```css
/* 수정된 CSS */
.ad-overlay {
    position: absolute;
    background: #000000;  /* 단색 검정 */
    opacity: 0;           /* 완전 투명으로 시작 */
}
```

```javascript
// JavaScript 제어
function setBackgroundImageOpacity(value) {
    const overlay = document.querySelector('.ad-overlay');
    // 반전: 100% 이미지 불투명도 = 0% 오버레이 불투명도
    overlay.style.opacity = (100 - value) / 100;
}
```

### 배운 교훈

**원칙: 전체 체인 테스트**

컨트롤을 구현할 때 UI 입력 → 상태 변경 → 시각적 결과까지 전체 파이프라인을 테스트할 것. 컨트롤이 "작동"(상태 변경)하면서도 실제로 보이는 결과를 생성하지 않을 수 있음.

## 3. 텍스트 오버플로우 이슈

### 문제

**날짜:** 2026년 2월  
**심각도:** 높음  
**영향:** AI 생성 텍스트가 캔버스 경계를 초과

**증상:**
"문구 생성" 버튼을 여러 번 클릭하면 헤드라인 텍스트가 점점 커지며 캔버스 경계를 넘어 오버플로우됨.

**근본 원인:**
생성 버튼에 중복 이벤트 리스너가 연결되어 있었음. 각 클릭이 핸들러를 두 번 트리거하여 누적 텍스트 증가 발생.

```javascript
// app.js (문제)
document.getElementById('ai-generate-btn').addEventListener('click', handler);

// figma-panel.js (역시 연결됨)
document.getElementById('ai-generate-btn').addEventListener('click', handler);

// 결과: 클릭당 핸들러가 두 번 호출됨
```

### 솔루션

중복 리스너를 제거하고 이벤트 처리를 한 곳에 통합.

```javascript
// figma-panel.js에서 단일 진실의 원천
init() {
    const generateBtn = document.getElementById('ai-generate-btn');
    if (generateBtn) {
        generateBtn.addEventListener('click', () => {
            this.generateContent();
        });
    }
}
```

### 배운 교훈

**원칙: 이벤트 바인딩의 단일 책임**

이벤트 리스너의 명확한 소유권 확립. 하나의 모듈이 주어진 엘리먼트에 이벤트를 바인딩하는 책임을 가져야 함. 어떤 모듈이 어떤 이벤트를 소유하는지 문서화.

## 4. 깊이 계층 이슈

### 문제

**날짜:** 2026년 2월  
**심각도:** 중간  
**영향:** UI가 평면적이고 비전문적으로 보임

**증상:**
컨트롤 패널, 카드, 입력 필드 모두 비슷한 배경색을 가져 시각적 깊이 계층 없이 평면적인 외관 생성.

**근본 원인:**
컬러 토큰이 정의되었지만 일관되게 적용되지 않음. 다른 컴포넌트가 의도된 계층을 따르지 않는 하드코딩된 색상 사용.

```css
/* 일관성 없는 적용 */
.panel { background: #0B0D12; }
.card { background: #0B0D12; }  /* 패널과 같음! */
.input { background: #0B0D12; } /* 카드와 같음! */
```

### 솔루션

각 레벨에 대해 뚜렷한 색상으로 명확한 깊이 계층 수립:

```css
:root {
    --bg-base: #2C2C2C;    /* 가장 깊음 - 캔버스 래퍼 */
    --bg-surface: #2C2C2C; /* 패널 배경 */
    --bg-input: #383838;   /* 입력 필드 - 더 밝음 */
}

/* 일관되게 적용 */
.control-panel { background: var(--bg-surface); }
.panel-section { background: var(--bg-surface); }
.figma-input { background: var(--bg-input); }
```

### 배운 교훈

**원칙: 디자인 토큰에는 규율이 필요**

토큰을 정의하는 것만으로는 충분하지 않음. 코드 리뷰와 린팅을 통해 사용을 강제할 것. 하드코딩된 색상을 완전히 피할 것.

## 5. 인트로 리디자인 결정

### 문제

**날짜:** 2026년 2월  
**심각도:** 낮음 (미적)  
**영향:** 첫인상이 제품 품질과 맞지 않음

**증상:**
인트로 스플래시 화면의 여러 반복이 원하는 "전문 크리에이티브 도구" 미학을 달성하지 못함. 디자인이 "이상해" 또는 기대와 맞지 않는다고 설명됨.

**근본 원인:**
디자인 방향이 불명확함. 반복이 명확한 참조점 없이 다른 스타일(3D 모션, 글래스모피즘, 정교한 애니메이션)을 시도함.

### 솔루션

본질적인 요소로 단순화:

```html
<div class="intro-splash">
    <div class="intro-content">
        <div class="intro-tagline">Build high-converting creatives with AI.</div>
        <img src="assets/Mefimake-logo.png" class="intro-logo-img">
        <div class="intro-loader">
            <span class="loader-dot"></span>
            <span class="loader-dot"></span>
            <span class="loader-dot"></span>
        </div>
    </div>
    <div class="intro-credit">2026 made by SEYOUNG LEE</div>
</div>
```

핵심 결정:
- 텍스트 대신 로고 이미지
- 명확한 가치 제안 태그라인
- 최소한의 애니메이션 (펄싱 도트)
- 최대 대비를 위한 검정 배경

### 배운 교훈

**원칙: 의심스러우면 단순화**

복잡한 애니메이션과 효과는 종종 가치보다 혼란을 추가함. 단순하게 시작하고 명확한 사용자 이점이 있을 때만 복잡성 추가.

## 6. 컬러 시스템 재균형

### 문제

**날짜:** 2026년 2월  
**심각도:** 중간  
**영향:** 애플리케이션 전체에서 일관성 없는 시각적 외관

**증상:**
사용자들이 UI에 "보라색 톤"이 나타난다고 보고함. 색상이 "뒤죽박죽" 느껴짐. 다크 테마가 응집력 있게 느껴지지 않음.

**근본 원인:**
여러 컬러 팔레트가 혼합됨:
- 초기 다크 테마 색상
- iOS 스타일 색상
- UI로 번지는 그라데이션 색상
- 하드코딩된 헥스 값

```css
/* 혼란을 일으키는 혼합 소스 */
background: #15141D;  /* 보라색 틴트 검정 */
background: #0F172A;  /* 네이비 */
background: #020617;  /* 거의 검정 */
background: #111827;  /* 그레이-블루 */
```

### 솔루션

iOS Dark Mode로 완전한 컬러 시스템 통일:

```css
:root {
    /* 단일 소스 팔레트 */
    --bg-base: #2C2C2C;
    --bg-input: #383838;
    --bg-input-hover: #262626;
    --bg-canvas: #1E1E1E;
    
    /* 모든 하드코딩된 색상 제거 */
    /* 네이비 톤을 중립 그레이로 교체 */
}
```

적용 방법:
1. 이전 컬러 값의 전역 검색/교체
2. 하드코딩된 색상에 대한 CSS 감사
3. 모든 화면에서 시각적 QA

### 배운 교훈

**원칙: 첫날부터 컬러 시스템**

UI 구축 전에 컬러 토큰을 수립하고 강제할 것. 소급적 컬러 변경은 비용이 많이 들고 오류가 발생하기 쉬움. 가능하면 컬러 린트 규칙 생성.

## 7. 초기 상태 버그

### 문제

**날짜:** 2026년 2월  
**심각도:** 중간  
**영향:** 새 사용자에게 일관성 없는 시작 상태

**증상:**
인트로 스플래시 후, 캔버스가 때때로 의도된 "Square" (1:1) 기본값 대신 "Story" (9:16) 사이즈를 표시함.

**근본 원인:**
`loadPlanFromStorage()` 함수가 `applyInitialState()`에서 설정한 초기 상태를 덮어씀.

```javascript
// 작업 순서
applyInitialState();      // Square 설정
loadPlanFromStorage();    // 저장된 상태(Story)로 덮어씀
```

### 솔루션

적절할 때 초기 상태를 존중하도록 `loadPlanFromStorage()` 수정:

```javascript
loadPlanFromStorage() {
    const saved = localStorage.getItem('mefimake_plan');
    if (saved) {
        // 사이즈가 아닌 프로퍼티만 복원
        // 또는 새로 시작 플래그 존중
    }
}

applyInitialState() {
    // 새 세션에는 항상 Square 사이즈 적용
    this.applyCanvasSize(1080, 1080, '1:1');
    
    // 1:1 카드를 활성으로 표시
    document.querySelector('[data-size="1:1"]').classList.add('active');
}
```

### 배운 교훈

**원칙: 명시적 초기화 순서**

초기화 시퀀스를 문서화하고 테스트할 것. 저장된 상태가 기본값을 덮어써야 하는지, 어떤 조건에서인지 고려.

## 8. 내보내기 변형 버그

### 문제

**날짜:** 2026년 2월  
**심각도:** 높음  
**영향:** 핵심 기능(전체 변형 내보내기)이 고장남

**증상:**
"전체 변형" 내보내기(15개 이미지)가 5개의 다른 콘텐츠 변형이 아닌 같은 이미지 5개의 복사본을 생성함.

**근본 원인:**
내보내기 함수가 캡처 사이에 실제로 콘텐츠를 변경하지 않고 현재 캔버스 상태를 5번 캡처함.

```javascript
// 문제가 있는 코드
async exportAllVariants() {
    for (let i = 0; i < 5; i++) {
        // 누락: this.applyVariant(i)
        await this.captureAndDownload();
    }
}
```

### 솔루션

생성 중에 5개 변형을 모두 저장한 다음, 캡처 전에 각각 명시적으로 적용:

```javascript
async exportAllVariants() {
    // 모든 변형 저장
    const variants = this.generatedVariants;
    
    for (let i = 0; i < variants.length; i++) {
        // 변형 콘텐츠 적용
        this.fillContentEditor(variants[i]);
        this.applyContentToCanvas();
        
        // 렌더링 대기
        await this.delay(200);
        
        // 각 사이즈 캡처
        for (const size of sizes) {
            await this.captureAtSize(size, i);
        }
    }
}
```

### 배운 교훈

**원칙: 복잡한 기능의 시각적 검증**

여러 출력을 생성하는 기능의 경우, 각 출력이 고유한지 수동으로 검증할 것. 자동화된 테스트에는 가능한 경우 시각적 diff 체크 포함.
